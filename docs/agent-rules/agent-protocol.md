# 프로토콜

이 문서는 모든 에이전트가 공통으로 따르는 행동, 기록, SSOT, 사용자 확인 규칙의 단일 출처(canonical source)다. `AGENTS.md`는 이 문서를 가리키는 summary layer다.

## 절대 원칙
- **우선순위:** 안전·보안 > 정확성 > 사용자 요청 > 단순함 > 저장소 관례 순으로 따른다.
- **작업 전 확인:** 이 문서의 원칙을 최우선으로 적용하며, 이후 관련 role guide와 구체적인 canonical source를 살핀다.
- **최소 변경:** 목적 달성에 필요한 최소한의 코드만 변경하며, 기존 스타일, 포매팅, 네이밍을 유지한다.
- **실행 전 확인:** 관련 코드, 설정, 문서, 테스트를 먼저 확인한다. 확인하지 않은 것을 확인했다고 거짓으로 말하지 않는다.
- **임의 판단 금지:** 문서 충돌, 보안, API 계약 등 결과에 영향이 큰 불확실성은 스스로 추측하지 말고 사용자에게 질문한다. 그 외는 가정을 명시하고 진행한다.
- **보안 및 승인:** 민감 정보는 환경 변수로 주입하며, 하단의 **[승인 필요]** 항목은 사용자 동의 없이 임의로 진행하지 않는다.

## 승인 필요 (사용자 확인 필수)
- 새 의존성 추가 또는 대규모 업그레이드
- DB 스키마 / 마이그레이션 변경
- 공개 API 계약 변경
- 파일 / 폴더 대규모 이동 또는 삭제
- 저장소 전체 포맷팅
- CI / CD / 인프라 설정 변경
- Git 이력 변경
- 환경 변수 / 운영 설정 변경
- 되돌리기 어렵거나 영향 범위가 넓은 변경

## 구현 / 변경
- 요청된 범위 안에서만 작업한다.
- 변경으로 인해 필요한 테스트, 문서, 설정의 최소 수정은 같은 작업 범위로 본다.
- 부분 수정(diff/patch) 우선, 전체 재작성은 꼭 필요할 때만 한다.
- 영향 범위가 넓은 변경은 바뀌는 동작과 유지되는 동작을 함께 보고한다.

## 실행 / 디버깅
- 작업을 검증 가능한 목표로 전환한다.
- 가능하면 작게 바꾸고 바로 검증한다.
- 같은 접근으로 3번 연속 실패하면 멈추고 원인과 다음 선택지를 정리한다.


## 테스트 / 완료 보고
- 퍼블릭 인터페이스와 기대 동작 중심으로 검증한다.
- 관련 테스트를 우선 실행하고, 이후 필요 시 범위를 넓힌다.
- 실행하지 못한 항목은 `미검증`으로 명확히 분리하여 보고한다.

## 문서 및 상태 관리 (docs/)
- **외부 앵커 유지 (Runtime Memory):** 컨텍스트 압축에 따른 유실과 세션 단절에 대비해, 현재 작업의 핵심 목표와 완료 기준을 `docs/memory/runtime/context.md`에 유지한다. 주요 판단 이력은 `dailylog.md`에 기록하되, 새 작업 세션 시작 시 이전 기록은 완전히 덮어써(Overwrite) 인지 과부하를 차단한다.
- **문서 파편화 억제 (Append-First & SSOT):** 새로운 `.md` 파일을 임의로 생성하지 않는다. 지식을 추가할 때는 가장 관련성 높은 기존 문서에 내용만 병합(Append/Update)한다. 중복 방지를 위해 구체적 내용은 단일 문서(SSOT)에만 남기고 타 문서는 링크로 참조한다. 신규 파일 생성이 불가피한 경우 반드시 사용자 승인을 받는다.
- **엄격한 지식 승격 (Promotion):** 1회성 트러블슈팅이나 지엽적인 버그 수정 내역은 승격 대상이 아니다. 작업 종료 시 '전역 범용 패턴', '설계 결정(ADR)', '치명적 실패 원인' 등 재사용성이 입증된 핵심 지식만 canonical source로 승격한다.
- **문서 무결성 검증:** `docs/` 하위 문서를 단 하나라도 수정했다면 링크 깨짐과 구조적 결함을 기계적으로 점검한다.
- **언어 및 표기 통제:** `docs/` 하위 문서는 기본적으로 `Korean`으로 작성한다. 단, AI의 임의 번역으로 인한 설정 오류를 막기 위해 코드, 파일 경로, 설정 키, 식별자 등은 절대 번역하지 않고 **원문(English) 표기**를 엄격히 유지한다.
